План переписывания PPTX-конвертера

Цель

Перейти от хрупких ключей вида s{slide}_sh{shape}_p{paragraph} к стабильным именам объектов PowerPoint из Selection Pane (cNvPr name). Основной ключ перевода будет жить в имени контейнера, а slideNumber и shapeId будут метаданными для человека, диагностики и fallback.

Файлы





[pptx-converter.js](pptx-converter.js): основная логика разбора, разметки, экспорта XLSX, импорта XLSX и отчёта предупреждений.



[pptx-convert.html](pptx-convert.html): добавить UI для первоначальной разметки PPTX и обновить подсказки на странице.



[readme-pptx.html](readme-pptx.html): обновить инструкцию, формат XLSX и ограничения.

Предлагаемый workflow





Пользователь загружает PPTX во вкладке “Prepare / Markup PPTX”.



Инструмент проходит по текстовым контейнерам, проставляет стабильные имена вида tr_<shortId>_<role> в cNvPr name и отдаёт размеченный PPTX.



Пользователь работает дальше только с размеченным PPTX.



Вкладка “PPTX → XLSX” экспортирует по одной строке XLSX на каждый непустой абзац PowerPoint (a:p), включая slideNumber, shapeId, containerType, paragraphIndex, исходный текст и языковые колонки.



Вкладка “XLSX → PPTX” ищет контейнеры прежде всего по shapeName, заменяет текст и показывает предупреждения по несовпадениям.

Формат XLSX

Колонки первого листа:





key: стабильный ключ абзаца, например tr_7f3a91_body__p0.



slideNumber: текущий номер слайда на момент экспорта, только для ориентации.



shapeName: имя объекта из Selection Pane, например tr_7f3a91_body.



shapeId: значение p:cNvPr/@id внутри PPTX, fallback в паре со слайдом.



containerType: shape, graphicFrame, возможно позже groupShape.



paragraphIndex: номер абзаца внутри контейнера.



<sourceLang>: исходный текст, например EN.



<targetLang>: перевод, например ES.

Пример XLSX-строки для одного абзаца:
tr_7f3a91_body__p0 | 2 | tr_7f3a91_body | 39 | shape | 0 | From founding... | Desde la fundacion...

Важно: это не визуальная строка на слайде. PowerPoint хранит текст блоками и абзацами (a:p); переносы внутри одного абзаца из-за ширины текстового поля не должны становиться отдельными единицами перевода.

Правила разметки имён





Размечаем все текстовые контейнеры с непустым текстом: p:sp и p:graphicFrame.



Если у контейнера уже есть валидное имя tr_<id>_<role>, сохраняем его.



Если имя отсутствует, дефолтное или дублируется, генерируем новое.



shortId генерируем как короткий случайный/детерминированный ID без номера слайда, например 7f3a91.



role определяем эвристикой:





title: крупный одиночный текст, верхняя зона слайда или placeholder title.



subtitle: одиночный текст рядом с title или placeholder subtitle.



body: обычный текстовый блок/абзац.



list: контейнер с несколькими абзацами или bullet-подобной структурой.



table: текст внутри p:graphicFrame с табличной структурой.



label: короткие подписи/бейджи/числа.



Роль нужна для читаемости, но не для стабильности. Стабильность обеспечивает tr_<id>.

Логика парсинга





Заменить текущий проход только по p:sp на общий collector: collectTextContainers(slideDoc).



Для каждого контейнера читать:





cNvPr id и name.



тип контейнера: shape или graphicFrame.



абзацы a:p и текстовые runs a:r/a:t.



placeholder metadata, если есть.



Поддержать graphicFrame, потому что в data/test-deck.pptx часть текста находится именно там, а текущий код её пропускает.

Логика вставки переводов





Основной матчинг: shapeName + paragraphIndex.



Если shapeName не найден: предупреждение missing-shape, без автоматической замены.



Если найдено несколько одинаковых shapeName: предупреждение duplicate-shape-name, без автоматической замены для неоднозначных строк.



Если shapeName найден, но shapeId или slideNumber отличаются от XLSX: заменить текст, но добавить warning уровня info/warn.



Если перевод пустой: оставить исходный текст без изменений и учитывать как skipped.



Форматирование оставить как сейчас: весь перевод класть в первый run абзаца, остальные runs очищать.

Ошибки и предупреждения

Показывать после каждой операции краткий отчёт рядом со ссылкой на скачивание:





сколько контейнеров размечено;



сколько строк экспортировано;



сколько переводов заменено;



сколько пропущено из-за пустого перевода;



сколько не найдено;



сколько дублей имён обнаружено;



список первых N предупреждений с возможностью скачать полный warning report как XLSX/CSV или JSON.

UI изменения





Добавить третью вкладку или первый шаг в текущую вкладку: “Prepare PPTX IDs”.



В pptx-convert.html обновить подсказки: ключи больше не позиционные, а основаны на Selection Pane names.



На вкладке импорта переименовать “Original presentation” в “Marked PPTX / presentation template”.



Добавить заметку, что после дублирования слайдов полезно снова прогнать normalization, чтобы исправить дубли tr_*.

Верификация





Проверить на data/test-deck.pptx, что экспортируется текст из p:sp и p:graphicFrame.



Проверить цикл: markup PPTX → extract XLSX → добавить перевод → inject PPTX.



Проверить сценарии предупреждений: удалённый shape, дублированный shapeName, пустой перевод, изменённый номер слайда.



Проверить, что PowerPoint открывает сгенерированные PPTX без repair prompt.
